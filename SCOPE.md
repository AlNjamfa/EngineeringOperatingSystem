# SCOPE

## Purpose

This document defines the boundaries of the Engineering Operating System (EOS).

Its purpose is to ensure the repository remains focused, useful, maintainable, and aligned with its mission throughout its lifetime.

Every future file, framework, template, case study, or idea should be evaluated against this document before being added.

The quality of the Engineering Operating System depends not only on what is included, but also on what is intentionally excluded.

---

# Mission Alignment

The Engineering Operating System exists to improve engineering judgment through disciplined questioning, structured reasoning, deliberate reflection, and continuous improvement.

Its objective is to help engineers make better decisions before, during, and after technical work.

Every addition to this repository must directly support that mission.

---

# Definition of the Project

The Engineering Operating System is a personal framework for systematic engineering thinking.

It is designed to help engineers:

* Define problems clearly.
* Understand business context.
* Challenge assumptions.
* Evaluate tradeoffs.
* Consider operational ownership.
* Think about FinOps and cost.
* Debug systematically.
* Reflect after implementation.
* Capture permanent lessons.
* Continuously improve engineering judgment.

The repository is intended to become a repeatable operating system for engineering decision-making rather than a collection of notes.

---

# What Belongs In This Repository

Content should be included only if it directly improves engineering judgment.

Examples include:

## Frameworks

Repeatable systems for approaching engineering problems.

Examples:

* Problem Solving
* Pre-Mortem
* Tradeoff Analysis
* Debugging
* After Action Review

---

## Question Libraries

Collections of high-quality questions that improve decision-making.

Examples:

* Engineering
* Business
* FinOps
* Leadership
* Communication

---

## Templates

Reusable documents that operationalize the frameworks.

Examples:

* Problem Session
* Case Study
* After Action Review

---

## Case Studies

Real examples demonstrating how the Engineering Operating System was applied to actual engineering work.

Case studies should focus on reasoning, tradeoffs, assumptions, lessons learned, and decision quality rather than technical complexity alone.

---

## Claude Guidance

Instructions that allow Claude to function as an Engineering Operating System partner by guiding structured thinking instead of immediately providing answers.

---

# What Does Not Belong

The following should not be included because they do not directly support the mission of the Engineering Operating System.

* Technology tutorials.
* Certification study guides.
* Command cheat sheets.
* Random notes.
* Personal journals unrelated to engineering judgment.
* Large collections of documentation copied from vendors.
* AI experiments unrelated to engineering thinking.
* Features added solely because they are interesting.
* Content that cannot reasonably be used during real engineering work.

This repository should remain focused on improving engineering judgment.

---

# Design Principles

Every part of the Engineering Operating System should follow these principles.

## Simplicity

Prefer fewer, higher-quality frameworks over many shallow documents.

Complexity should only be added when it clearly improves engineering judgment.

---

## Reusability

Every framework should apply across technologies whenever possible.

The frameworks should remain useful whether working with:

* AWS
* Terraform
* Docker
* Kubernetes
* CI/CD
* Networking
* Linux
* FinOps
* Future technologies

Technology changes.

Good engineering thinking does not.

---

## Practicality

Every framework should be usable during a real engineering task.

If a framework cannot realistically be used during a Jira ticket, project, or incident, it should be reconsidered.

---

## Output-Oriented Thinking

Every framework should produce a tangible output.

Examples include:

* Problem statement
* Decision record
* Risk assessment
* Validation plan
* Root cause analysis
* Lesson learned
* Case study

Thinking should produce artifacts.

Artifacts should improve future thinking.

---

## Continuous Improvement

The Engineering Operating System is a living system.

Real engineering experience should improve the frameworks over time.

Every completed project should strengthen the Engineering Operating System.

---

# Definition of Done

A new framework, template, or document is complete only if all of the following are true:

* It improves engineering judgment.
* It supports the mission of the repository.
* It is reusable.
* It is practical during real engineering work.
* It produces a meaningful output.
* It avoids unnecessary complexity.
* It would make sense to another engineer reading it.
* It demonstrates mature engineering reasoning.
* It could be followed by Claude as a structured process.

If any of these conditions are not met, the work is not complete.

---

# Scope Validation Questions

Before adding anything new, ask:

* Does this improve engineering judgment?
* Will I actually use this during a real engineering task?
* Does this strengthen systematic thinking?
* Does this belong in an Engineering Operating System rather than a notes repository?
* Could this be incorporated into an existing framework instead?
* Does this reduce future mistakes?
* Does this help create stronger case studies?
* Would a hiring manager understand why this exists?
* Does this align with the mission of the repository?

If the answer to any of these questions is "no," reconsider adding it.

---

# Long-Term Vision

The Engineering Operating System should become more valuable every year.

Its growth should come from better questions, stronger frameworks, richer case studies, and improved engineering judgment rather than an increasing number of files.

The ultimate measure of success is not the size of the repository.

The ultimate measure of success is whether it helps create engineers who consistently make better decisions.

---

# Final Principle

The Engineering Operating System exists to improve engineering judgment.

It is intentionally designed to remain focused.

Every addition should strengthen the system.

Nothing should be added that does not directly help an engineer think more clearly, make better decisions, or learn more effectively from experience.

The purpose of this repository is not to collect information.

The purpose of this repository is to cultivate engineering wisdom.
